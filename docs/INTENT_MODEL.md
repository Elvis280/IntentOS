# IntentOS: Intent Model Specification v1.0

## 1. Product Philosophy

Traditional software operates on a paradigm of physical manipulation: users must understand complex, application-specific interfaces to execute rigid sequences of clicks, keystrokes, and menu traversals. Automation tools replicate this same flaw, requiring users to explicitly program mechanical workflows that break when pixels shift.

**IntentOS fundamentally rejects this model.**

IntentOS operates on the principle that the user should express **what they want** (the outcome), while the system determines **how it should be achieved** (the execution). This decoupling of *Intent* from *Execution* is the defining architectural philosophy of the platform.

Whether the user wants to "make the title darker" in PowerPoint, "refactor this class" in VS Code, or "book the cheapest flight" in Chrome, the underlying application interface changes completely—but the interaction model does not. The user commands outcomes; IntentOS manages the interface.

## 2. Core Definitions

To ensure architectural clarity, IntentOS strictly separates the cognitive hierarchy of work:

* **Intent:** The overarching expression of a desired outcome (e.g., "Make this presentation look professional"). Intents are abstract, persistent, and outcome-oriented.
* **Goal:** A specific, measurable milestone within an Intent (e.g., "Standardize all slide fonts to Arial").
* **Objective:** A logical phase required to reach a Goal (e.g., "Identify current font configurations").
* **Task:** A discrete block of executable work generated by the Reasoner (e.g., "Select all text on Slide 1").
* **Command:** A rigid OS or application-level directive (e.g., `CTRL+A`).
* **Action:** The atomic physical primitive executed by the system (e.g., `CLICK(x, y)` or `TYPE("Hello")`).
* **Instruction:** The raw natural language input provided by the user.
* **Workspace:** The aggregate collection of all open applications, windows, and resources currently active on the OS.
* **Context:** The semantic understanding of what is currently happening within the Workspace (e.g., "The user is editing a financial spreadsheet").
* **Outcome:** The final, verified state of the Workspace compared against the original Intent.
* **Session:** The continuous, long-lived collaboration window during which Intents are formulated, evolved, and resolved.

## 3. Intent Hierarchy

Intents are not flat. They exist on a spectrum of abstraction, and the Runtime is responsible for decomposing higher-level intent into smaller executable logic without exposing the mechanics to the user.

* **Strategic Intent (The Destination):** 
  * High-level, ambiguous, spanning multiple applications and sessions. 
  * *Example:* "Prepare my quarterly review presentation."
* **Operational Intent (The Strategy):** 
  * Medium-level, focused on specific phases of work within a known application context. 
  * *Example:* "Add three slides summarizing Q3 revenue."
* **Micro Intent (The Tweak):** 
  * Low-level, immediate corrections or precise adjustments. 
  * *Example:* "Move that chart slightly to the left," or "Make the text bold."

## 4. Intent Characteristics

An Intent is a complex data structure, not a mere text string. An Intent possesses:
* **Purpose:** The fundamental desired outcome.
* **Priority:** Execution urgency relative to other Intents.
* **Scope:** The boundary of affected applications (e.g., isolated to Excel, or crossing to Chrome).
* **Target:** The specific UI elements or data objects affected.
* **Constraints:** Hard limits (e.g., "Do not delete any existing data").
* **Expected Outcome:** The verifiable state change.
* **Confidence:** The Runtime's certainty in its understanding of the Intent.
* **Current Status:** e.g., Pending, Evolving, Executing, Verified.
* **Dependencies:** Pre-requisite states (e.g., "Requires a browser window to be open").

## 5. Intent Context

An Intent cannot be resolved in a vacuum. The statement "Close this" means entirely different things depending on context. Every Intent is evaluated dynamically against:
* **Current Workspace:** The physical arrangement of windows and monitors.
* **Current Session:** The ongoing history of the user's collaboration.
* **Current World:** The spatial mapping of UI elements at this exact millisecond.
* **Current Application:** The specific software in focus.
* **User History:** Past behaviors and preferences.
* **Current Selection:** What the user has highlighted with their mouse or keyboard.
* **Previous Intents:** The trajectory of the conversation.

## 6. Intent Resolution Flow

Transforming an abstract Intent into atomic execution involves a strict conceptual pipeline:

```
User Intent
    ↓
Context Understanding (Binding the Intent to the current Session/Workspace)
    ↓
Workspace Analysis (Binding the Intent to physical UI coordinates)
    ↓
Reasoning (Generating an execution strategy)
    ↓
Policy Validation (Ensuring the strategy is safe and optimal)
    ↓
Skill Selection (Swapping raw AI clicks for robust, hardcoded macros if possible)
    ↓
Execution (Triggering OS interactions)
    ↓
Verification (Confirming the Workspace state matches the Expected Outcome)
    ↓
Intent Updated (Marked as resolved, or queued for refinement)
```

## 7. Intent Evolution & Human Collaboration

Intents are fluid. Users rarely provide a perfect, comprehensive instruction upfront. They refine their desires as they see the work unfold.

An Intent must evolve seamlessly:
1. *"Create a presentation."* (Initial Intent)
2. *"Use a dark theme."* (Intent Refined)
3. *"Replace the images."* (Intent Evolved)
4. *"Reduce the text."* (Intent Iterated)

The system does not restart execution for each statement. It evolves the overarching Intent, updating the context and applying the new constraints to the ongoing Session. 

**Collaboration is fundamental.** The user must always have the ability to interrupt execution, modify the Intent mid-flight, approve or reject proposed changes, undo actions, and continue working manually without breaking the IntentOS context.

## 8. Intent Persistence (Sessions vs Jobs)

* **Jobs** are transient execution units. They succeed, fail, and terminate.
* **Sessions** are long-lived collaborative windows. 
* **Intents** live within Sessions, not Jobs.

An Intent may spawn multiple Jobs over time. An Intent can be paused while the user works manually, resumed an hour later, refined over days, and eventually archived into Memory. 

## 9. Relationship to Other Systems

The Intent Model dictates the behavior of all other architectural components:
* **Runtime:** Orchestrates the lifecycle of the Intent from creation to resolution.
* **World Model:** Provides the physical canvas upon which the Intent is executed.
* **Context Engine:** Provides the semantic canvas upon which the Intent is understood.
* **Reasoner:** Decomposes the Intent into Tasks and Actions.
* **Policy Engine:** Enforces safety constraints on how the Intent is achieved.
* **Skills:** Provide deterministic shortcuts to achieve common Intents safely.
* **Executor:** Physically manifests the Intent.
* **Verifier:** Confirms the Intent was successfully manifested.
* **Memory:** Archives resolved Intents to inform future Context.
* **Reflection:** Analyzes failed Intents to improve future Reasoning.
* **Recovery:** Salvages failing execution paths to protect the overarching Intent.

## 10. Future Evolution

The Intent Model is designed to scale horizontally across new modalities without requiring architectural redesigns:
* **Voice Interaction:** Voice is simply another transport layer for generating Intents.
* **Vision-Guided Interaction:** A user circling an element with an Apple Pencil generates an Intent with a highly specific spatial Constraint.
* **Multi-Step Collaboration:** Complex Intents can be paused, handed back to the user for manual data entry, and resumed automatically.
* **Application-Specific Plugins:** Specialized modules that allow the Reasoner to bypass physical UI manipulation and resolve Intents via direct API injection.
* **Personalization:** Memory adjusts how abstract Intents ("Make it look good") are resolved based on historical user preferences.

## 11. Design Principles

Every engineering decision regarding intent processing must adhere to these tenets:
1. **Intent over interface:** Never force the user to describe *how* to click a button.
2. **Outcomes over procedures:** The system is judged by the final state of the desktop, not the elegance of its clicks.
3. **Context before execution:** An Intent executed blindly without workspace context is dangerous.
4. **Human collaboration over blind autonomy:** The user is the pilot; IntentOS is the co-pilot.
5. **Deterministic execution whenever possible:** Fulfill Intents using robust Skills, relying on generative AI execution only as a fallback.
6. **Continuous refinement over one-shot automation:** Embrace ambiguity and allow the user to steer execution organically.
7. **Explicit state over hidden behavior:** The user must always know how the system has interpreted their Intent.
8. **The user describes the destination; IntentOS determines the safest and most reliable path.**
